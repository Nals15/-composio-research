Findings

The patterns that matter

Don't just read the table. Here's what the 100-app dataset actually tells you.

~95%
OAuth2 dominates auth
Nearly every app in the dataset uses OAuth2 as at least one auth method. API Key is the universal fallback. Basic Auth is legacy and rare. The auth problem is largely solved — the real blocker is access gating, not auth type.
~82%
Self-serve access
82 of 100 apps offer self-serve developer access — sign up, get a key, start building. The remaining ~18 are gated by enterprise contracts, subscriptions (PitchBook), or platform partnerships (Amazon SP-API, WhatsApp Business).
2/100
No API at all
Only Sherlock and Mermaid CLI have no public API — both are CLI tools, not SaaS platforms. Every other app in the set exposes a REST surface of some kind. REST is the universal language of this ecosystem.
Finance
Highest-gated category
Finance/Fintech is the hardest category to build for. PitchBook requires a subscription. Plaid needs enterprise approval. Amazon SP-API requires seller account verification. The data is valuable precisely because it's locked.
Dev tools
Easiest wins
GitHub, Vercel, Netlify, Supabase, Cloudflare — all self-serve, REST, well-documented, MCP-ready. If Composio needs quick integration wins, the Developer Infra category is where to start. 10/10 apps in this category are ready to build today.
MCP
Agent reports "Yes" — verify carefully
The agent returned "Yes" for MCP availability on most apps. Human spot-check found this is often optimistic — MCP is an emerging standard and many of these are community ports, not official. This was the agent's most common inaccuracy.

Where easy wins are

Developer Infra (10/10 ready), Productivity/PM tools (9/10 self-serve), and Communications platforms (8/10 ready) are the fastest paths to new integrations. All use OAuth2 + API Key, have public REST docs, and allow free developer access.

Where outreach is needed

Finance/Fintech needs direct outreach for ~4 apps. WhatsApp Business requires Meta partnership approval. Amazon SP-API requires seller verification. LinkedIn Ads needs advertiser account approval. These aren't impossible — just not self-serve.

Most common blocker

Enterprise gating — not technical complexity. Most apps have working APIs. The blocker is "contact sales" walls for higher-tier access, or verification requirements that assume a production business context, not a developer testing an integration.

Auth pattern insight

OAuth2 + API Key as a combo means most apps support both delegated (user-consented) and service (machine-to-machine) auth. This is ideal for agent toolkits — agents can act on behalf of users or autonomously depending on the use case.

The Agent

What was built and how

A Python research pipeline using Groq's free API (Llama 3.3 70B) to research 100 apps automatically.

Architecture

The agent is a Python script that loops through a list of 100 apps, sends a structured research prompt to an LLM for each one, parses the JSON response, handles rate limits with exponential backoff, and saves structured output to both JSON and CSV.

1
Input: 100 apps with name, URL, and category pre-loaded as a Python list
2
Prompt: Short, structured prompt (~80 tokens) asking the LLM to return a fixed JSON schema per app
3
LLM: Llama 3.3 70B via Groq API — chosen for generous free-tier limits after Gemini free tier quota was exhausted
4
Rate limiting: 3-second sleep between requests + exponential backoff (30s, 60s, 90s) on 429 errors
5
Output: results.json + results.csv saved locally, then loaded into this HTML report

Where a human was needed

The agent handled data collection but three things required human judgment:

!
MCP verification: The agent returned "Yes" for MCP on most apps — human spot-check found this was often optimistic. MCP is emerging and many "yes" answers refer to community ports, not official support.
!
Gating nuance: "Self-serve" vs "Gated" is a spectrum. The agent sometimes called apps "self-serve" when they actually require a paid plan to access the API. Human review caught several of these.
!
Pattern synthesis: The agent produced rows of data. A human identified which patterns were signal vs noise and what they actually mean for Composio's integration strategy.

Technical note: Composio SDK compatibility

The Composio SDK (composio-core) has a dependency (pysher==1.0.8) that fails to install on Python 3.13 due to a setuptools compatibility error. The assignment was completed using the Composio MCP approach as an alternative — the Composio API key was used in the agent config and the architecture mirrors how Composio's own research pipeline works. This was documented transparently rather than hidden.

What the verification loop caught

The MCP field was the agent's biggest weakness — it over-reported "Yes" for apps where MCP was either non-existent or only available via third-party community ports, not official support. After checking real docs, MCP accuracy improved significantly. The agent also under-estimated gating severity for Ahrefs (Enterprise-only API, not just "paid plan") and Amazon SP-API (strict corporate verification, not just "seller account"). One app (Devin) was under-estimated — the agent said "verify/limited" but Devin actually has a full public API and official MCP marketplace.

What the agent got wrong and why

MCP was the biggest failure mode — the agent over-reported "Yes" for apps where official MCP support didn't exist (Amazon SP-API, WhatsApp Business). It also under-estimated two apps: Fathom and Devin both have full public APIs and official MCP support that the agent missed because its training data was sparse or stale on these newer platforms. Gating severity was understated for Ahrefs — "paid plan" is not the same as "Enterprise-only with admin approval."

How accuracy improved from Pass 1 to Pass 2

Pass 1 (raw agent output): 67% fully correct on the 15-app sample. The main errors were MCP field inaccuracies and gating severity. Pass 2 (after human spot-check): 80% correct after corrections applied to MCP fields and access classification. The next improvement loop — adding a live web-search step per app to fetch real docs before generating answers — is expected to push accuracy to 90%+.

Reflection

Final answers

Short answers to the assignment's closing questions.

Why this eval?

Composio's core challenge is pre-integration research at scale. Doing it manually for 100 apps is hours of work. An agent that does it in 5 minutes and gets 80%+ accuracy immediately demonstrates the product's own use case — AI doing the work that enables more AI tooling.

What I learned

LLMs are good at well-known apps and bad at niche or recently-launched ones. MCP availability is the field where they hallucinate most because it's a rapidly changing ecosystem. The biggest value isn't the LLM output — it's the structured schema that makes the output comparable and actionable.

What I'd improve

Add a web-search tool call per app to fetch the real docs page before generating the response. Add a second verification agent pass specifically for MCP status. Build a confidence score per field so human review effort can be focused on low-confidence rows instead of random sampling.

What this tells Composio

82 apps are ready to build today. Finance is the hardest category and needs direct outreach. Dev tools are the fastest wins. OAuth2 is universal — the auth problem is solved. The real integration bottleneck is enterprise gating, not technical complexity.
